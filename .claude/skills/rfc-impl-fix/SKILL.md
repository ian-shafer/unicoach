# Skill: RFC Implementation Fix

This skill takes the output of an `rfc-impl-review` report and acts as the
strict "fixer." It implements the explicitly recommended options from the
review's "Action Items for Implementor" section.

**Role — worker.** Wears the **worker** hat from `iterative-work`: one fix pass,
honours its write-scope allowlist, records a per-finding ledger to its scratch
sub-path (write-once, skip-if-present), never commits/stashes. The general rules
live there; below is the fix-specific flow.

## Critical Behaviors

- **Context Inheritance**: You execute within the same context as the original
  `rfc-impl` skill. You must ensure you have ingested the target RFC, any
  touched directory's `INVARIANTS.md`, and required defensive coding skills
  before modifying code.
- **Strict Adherence**: You MUST strictly implement only the fixes recommended
  in the report. However, the target RFC remains the absolute ground truth:
  before applying any recommendation, verify that it does not contradict or
  exceed the scope of the target RFC. If a recommendation deviates from or
  violates the RFC, you MUST reject it and report this discrepancy. Do NOT
  introduce speculative changes, extra refactoring, or feature creep.
- **NO GIT COMMITS**: You MUST NOT make any git commits. Leave all modifications
  in the local working directory. The architect or external review skills will
  evaluate the raw uncommitted diffs.
- **Verification**: After applying the fixes, you MUST run relevant tests (e.g.,
  `bin/test` or specific test suites) to ensure your changes compile and pass.

## Execution Flow

1. **Read the Report**: Identify and ingest the "Master Implementation Review"
   report generated by `rfc-impl-review`. If the report is not explicitly
   provided in the context, ask the user to provide it.
2. **Apply Action Items**: Parse the "Action Items for Implementor" section.
   When the caller supplied a **run-scratch sub-path**, maintain a per-finding
   ledger at `<scratch>/ledger.jsonl` — one line per action item recording its
   id and outcome (`applied` | `skipped` | `failed`). For each item, **in
   order**:
   - **Skip-if-present:** if the ledger already records this item as `applied`
     or `skipped` (a prior run handled it), skip it. This is what lets a fix
     pass that stalled partway through **resume at the first unapplied finding**
     rather than re-doing — or double-applying — earlier ones.
   - Review the provided options; you MUST select and implement the explicitly
     recommended option. If a recommendation requires clarity, use your codebase
     search tools to verify context before modifying files.
   - **Append the ledger line the instant the item is resolved** (write-once),
     before moving to the next item — so the work is captured even if the pass
     is interrupted mid-stream.
3. **Execute Tests**: Verify the implementation code compiles and all related
   test suites pass. If tests fail, systematically debug and fix them based only
   on the boundaries of the action items.
4. **Demonstrate Success**: Upon completing the fixes and verifying a green test
   suite, explicitly inform the architect that the fixes are complete and ready
   for their review or subsequent automated verification cycles.
