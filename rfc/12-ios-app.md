# iOS Application Registration Spec

## Executive Summary

This specification defines the architecture and implementation plan for an iOS
application limited strictly to registration, omitting the login and
password-reset workflows, built to register users with the Unicoach REST HTTP
server. The client will be authored in Swift using SwiftUI for the user
interface, and utilize the native `URLSession` networking stack to interface
with the `/api/v1/auth/register` endpoint.

## Detailed Design

### Toolchain and Dependencies

The application requires a minimum deployment target of **iOS 17.0** and **Swift
6.0**. It relies exclusively on native standard libraries (SwiftUI, Foundation)
with zero third-party dependencies. To allow for local integration testing
against the backend REST server, the project's `Info.plist` will explicitly
configure `NSAppTransportSecurity` to permit local networking over plain HTTP.

### Architecture Pattern

The app will enforce a strict **MVVM (Model-View-ViewModel)** architecture:

- **View**: A SwiftUI `View` containing the registration form (email, password,
  display name). TextFields will actively use view modifiers to disable
  autocorrection, autocapitalization, and force the `.emailAddress` keyboard
  style. All UI elements will explicitly set `.accessibilityIdentifier` and
  `.accessibilityLabel` properties for robust UI testing. Form navigation will
  be strictly controlled via `@FocusState` to naturally advance from Email ->
  Name -> Password fields on keyboard "Return". The submit button will be
  programmatically disabled during inflight requests to prevent duplication.
- **ViewModel**: An `ObservableObject` explicitly marked with `@MainActor` to
  enforce strict concurrency safety. It handles local validation (e.g., ensuring
  passwords meet the minimum 8-character backend boundary before issuing a
  network call) and safely mutating bound loading/error states on the main
  thread. It maps any `fieldErrors` returned into a local state to highlight
  specific failing TextFields dynamically using localized strings
  (`String(localized:)`).
- **Model / Networking**: A network layer managing standard Foundation
  `URLSession` calls utilizing `os.Logger` (`import os`) to strictly trace
  lifecycle execution. Models will strictly map to schemas. The backing
  `AuthClient` service must accept an injected base URL configuration (e.g.,
  from Plist/Env) so the host is dynamically resolved per environment.

### API Contracts and Network Data Models

The client will interface with the `POST /api/v1/auth/register` endpoint. The
network layer will encode payloads exclusively using `JSONEncoder()` and firmly
append the `Content-Type: application/json` HTTP request header to circumvent
standard 415 backend rejections. JSON payloads adhere to OpenAPI 3 structures:

```swift
struct RegisterRequest: Codable {
    let email: String
    let password: String
    let name: String
}

struct PublicUser: Codable {
    let id: UUID
    let email: String
    let name: String
}

struct RegisterResponse: Codable {
    let user: PublicUser
}

struct FieldError: Codable {
    let field: String
    let message: String
}

struct ErrorResponse: Codable, Error, Identifiable {
    var id: String { code }
    let code: String
    let message: String
    let fieldErrors: [FieldError]?
}
```

### Session Persistence

Cookie management is inherently stateful handled automatically by leveraging the
native `URLSessionConfiguration.default`. The `URLSession` will intercept the
`HttpOnly` and `Secure` `Set-Cookie` headers responding from the backend and
will persist them via `HTTPCookieStorage.shared`. Future HTTP executions will
automatically attach these session cookies natively.

### Error Handling and Edge Cases

The `URLSessionConfiguration` will be explicitly configured with a
`timeoutIntervalForRequest` of 10 seconds. When encountering network timeouts,
the application will natively generate a synthetic timeout struct (e.g.,
`ErrorResponse(code: "TIMEOUT", message: String(localized: "Network request timed out."))`).
When encountering HTTP 400, 409, or 500 status codes, the client will wrap the
`JSONDecoder().decode()` execution inside a `do-catch` block. If the payload
fails to parse cleanly as JSON (e.g., encountering raw HTML generated by a 502
Bad Gateway proxy failure), the application will safely swallow the
`DecodingError` and map it to a generic `.alert(item:)` overlay rather than
crashing abruptly during string interpolation.

## Tests

Verification will be handled via standard **XCTest** unit tests. The testing
scope includes:

- Asserting the `RegistrationViewModel` correctly transitions input validation
  states (e.g., rejecting invalid emails locally without initiating network
  calls).
- Asserting the `RegistrationViewModel` securely toggles its loading boolean
  properties across asynchronous await boundaries.
- Mocking the network pipeline via a custom `URLProtocol`. Test suites will
  strictly instantiate an ephemeral `URLSessionConfiguration` and inject the
  custom mock protocol to explicitly isolate network testing from the global
  `HTTPCookieStorage` or local caches.
- Validating that `URLSession` correctly encodes the `RegisterRequest` data
  structure, including HTTP header assertions.
- Asserting that HTTP 201 successful responses accurately map correctly back
  into the `RegisterResponse` construct.
- Asserting HTTP 4xx/5xx status status mappings accurately surface an
  `ErrorResponse`.

## Implementation Plan

1. **Initialize Project Skeleton**: Generate the `.xcodeproj` file alongside the
   foundational `@main` application entry point (`UnicoachiOSApp.swift`) and an
   empty `Assets.xcassets` catalog.
   - _Verify_: Run
     `xcodebuild -workspace ios-app/UnicoachiOS.xcodeproj/project.xcworkspace -scheme UnicoachiOS build`
     and ensure no build failures.
2. **Implement Network Contracts**: Define the JSON codable models
   (`Models.swift`).
   - _Verify_: Compile project to ensure models are structurally sound.
3. **Implement Network Client**: Build `AuthClient.swift` strictly isolating the
   `URLSession.shared.data(for:)` function call and managing error mapping,
   explicitly setting the 10-second timeout.
   - _Verify_: Compile project to ensure `AuthClient` passes type checks.
4. **Implement ViewModel**: Build `RegistrationViewModel.swift` configuring
   `@Published` properties for input fields, loading indicator, and error
   messaging.
   - _Verify_: Compile project to ensure `ViewModel` binds correctly to
     dependencies.
5. **Implement User Interface**: Build `RegistrationView.swift` containing
   standard SwiftUI `TextField`, `SecureField`, and `.alert()` views binding
   directly to the ViewModel.
   - _Verify_: Run the SwiftUI Preview or `xcodebuild` to verify view rendering
     logic is valid.
6. **Implement Test Suites**: Build `AuthClientTests.swift` to verify synthetic
   network responses and `RegistrationViewModelTests.swift` to assert structural
   ViewModel input validations.
   - _Verify_: Run
     `xcodebuild test -workspace ios-app/UnicoachiOS.xcodeproj/project.xcworkspace -scheme UnicoachiOS -destination 'platform=iOS Simulator,name=iPhone 16'`
     ensuring all assertions pass cleanly.
